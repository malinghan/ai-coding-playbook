## 核心原则

### 1. 明确 > 模糊

AI 不会猜你的意图，越具体越好：

```
# 模糊
"优化这个函数"

# 明确
"优化 process_orders() 函数的性能：
- 当前瓶颈是循环内的数据库查询（N+1 问题）
- 用批量查询替代，保持函数签名不变
- 不改变返回值结构"
```

### 2. 约束 > 自由发挥

告诉 AI 不能做什么，和告诉它要做什么同样重要：

```
"重构 UserService 类：
- 不改变公共方法签名（其他模块依赖这些接口）
- 不添加新的依赖库
- 不修改数据库 schema
- 只关注 src/services/user.ts，不动其他文件"
```

### 3. 先计划，再执行

对于复杂任务，让 AI 先输出计划，确认后再执行：

```
"在修改任何代码之前，先列出：
1. 需要修改哪些文件
2. 每个文件的具体改动
3. 可能影响的其他模块
4. 需要更新的测试

等我确认后再开始"
```

### 4. 分步骤 > 一次性

把大任务拆成小步骤，每步确认后再继续：

```
# 不好：一次性要求太多
"重构整个认证模块，添加 OAuth，更新测试，写文档"

# 好：分步骤
第一步："分析 src/auth/ 目录，列出现有结构和问题"
第二步："重构 login.ts，把回调改为 async/await"
第三步："添加 Google OAuth 支持"
第四步："更新测试"
```

## 任务描述技巧

### 给出背景和目标

```
"背景：我们的 API 响应时间从 200ms 增加到了 2s，
通过 profiling 发现瓶颈在 getUserWithPermissions() 函数。

目标：把这个函数的执行时间降到 300ms 以内。

约束：不能改变函数签名，不能破坏现有测试。"
```

### 指定技术栈和版本

```
"用 React 18 + TypeScript 5 实现一个分页组件：
- 使用 React Query v5（不用 SWR）
- 样式用 Tailwind CSS（不用 CSS Modules）
- 支持受控和非受控两种模式"
```

### 提供示例输入输出

```
"实现一个函数 parseDate(input: string): Date | null

示例：
- parseDate('2024-01-15') → Date(2024, 0, 15)
- parseDate('15/01/2024') → Date(2024, 0, 15)
- parseDate('Jan 15, 2024') → Date(2024, 0, 15)
- parseDate('invalid') → null
- parseDate('') → null"
```

### 引用现有代码作为参考

```
"参考 src/api/users.ts 的实现风格，
为 teams 资源实现相同的 CRUD 接口。
保持相同的错误处理模式和响应格式。"
```

## 上下文管理

### 精确指定相关文件

不要让 AI 猜，主动告诉它需要看哪些文件：

```
# Claude Code
"读取 src/auth/login.ts 和 src/middleware/auth.ts，
然后修复 token 刷新的竞态条件"

# Cursor
"@src/auth/login.ts @src/middleware/auth.ts
修复 token 刷新的竞态条件"

# Aider
/add src/auth/login.ts src/middleware/auth.ts
"修复 token 刷新的竞态条件"
```

### 用只读文件提供规范

```
# 把规范文档作为参考，不让 AI 修改它
/read-only docs/api-design-guide.md  # Aider
@Docs/API规范                         # Cursor

"按照 api-design-guide.md 中的规范实现新接口"
```

### 控制上下文大小

上下文越大，AI 越容易分心：

```
# 不好：把整个项目都加进来
"分析整个 src/ 目录，找出所有问题"

# 好：精确指定范围
"只分析 src/payments/ 目录，找出潜在的并发问题"
```

## 复杂任务拆解

### TDD 驱动开发

```
"用 TDD 方式实现邮件发送功能：

第一步：写测试用例，覆盖：
- 正常发送（单个收件人）
- 批量发送（多个收件人）
- 发送失败重试（最多 3 次）
- 无效邮箱地址处理

第二步：运行测试，确认全部失败（红灯）

第三步：实现最小可用代码让测试通过（绿灯）

第四步：重构代码，保持测试通过（重构）"
```

### 渐进式重构

```
# 第一步：不改变行为，只改结构
"把 UserController 中超过 50 行的方法提取到独立函数，
不改变任何业务逻辑，只做结构调整"

# 确认后，第二步：改进逻辑
"现在优化提取出来的 validateUserInput() 函数，
添加更完整的验证规则"
```

### 探索后再实现

```
# 先探索，不修改代码
"分析 src/cache/ 目录的现有实现，告诉我：
1. 当前缓存策略是什么
2. 有哪些已知问题
3. 你建议如何改进

不要修改任何代码，只分析和建议"

# 确认方向后再实现
"按照你刚才的建议，实现 LRU 缓存替换方案"
```

## 代码审查与调试

### 让 AI 解释后再修改

```
"先解释 src/queue/processor.ts 中的重试逻辑，
特别是指数退避的实现，确认我理解正确后再优化"
```

### 提供错误上下文

```
"运行 npm test 后出现以下错误：

[粘贴完整错误信息]

相关代码在 src/auth/login.ts:45-80。
分析根本原因并修复，不要只修复症状。"
```

### 要求解释修改原因

```
"修复这个 bug，并解释：
1. 为什么会出现这个问题
2. 你的修复方案是什么
3. 有没有其他可能的修复方式，为什么选这个"
```

### 安全审查

```
"审查 src/api/upload.ts 的安全性，重点检查：
- 文件类型验证是否可以被绕过
- 文件大小限制是否有效
- 上传路径是否有目录遍历风险
- 是否有 SSRF 风险

只列出问题，不要修改代码"
```

## 常用提示词模板

### 实现新功能

```
"实现 [功能名称]：

需求：
- [具体需求 1]
- [具体需求 2]

技术约束：
- 使用 [技术栈]
- 不使用 [禁止的库/方法]
- 保持与 [参考文件] 相同的风格

验收标准：
- [可测试的标准 1]
- [可测试的标准 2]"
```

### 修复 Bug

```
"修复 [描述问题]：

复现步骤：
1. [步骤 1]
2. [步骤 2]

期望行为：[应该发生什么]
实际行为：[实际发生什么]

相关文件：[文件路径]
错误信息：[完整错误信息]

修复要求：
- 找出根本原因，不只修复症状
- 不改变函数签名
- 添加防止回归的测试"
```

### 代码重构

```
"重构 [文件/函数]：

目标：[重构的目的，如提高可读性/性能/可测试性]

约束：
- 不改变公共 API
- 保持现有测试通过
- 不添加新功能
- 只修改 [指定范围]

完成后说明：
- 做了哪些改动
- 为什么这样改"
```

### 生成测试

```
"为 [函数/模块] 生成测试：

测试框架：[Vitest/Jest/pytest 等]
覆盖范围：
- 正常流程（happy path）
- 边界条件（空值、最大值、最小值）
- 错误情况（无效输入、网络失败等）
- 并发场景（如果适用）

要求：
- Mock 所有外部依赖
- 每个测试只验证一件事
- 测试描述要清晰说明测试什么"
```

### 性能优化

```
"优化 [函数/模块] 的性能：

当前问题：[描述性能瓶颈，如 N+1 查询、内存泄漏等]
性能目标：[具体指标，如响应时间 < 100ms]

约束：
- 不改变函数签名和返回值
- 不破坏现有测试
- 优先考虑可读性，只在必要时牺牲可读性换性能

请先分析根本原因，再提出优化方案，等我确认后再实现"
```

## 反模式（避免这些）

### 过于宽泛的指令

```
# 不好
"优化代码"
"修复 bug"
"重构这个模块"

# 好
"优化 getUserList() 的数据库查询，解决 N+1 问题"
"修复用户登录后 JWT token 没有设置到 cookie 的 bug"
"把 UserModule 中的业务逻辑从 controller 移到 service 层"
```

### 一次要求太多

```
# 不好
"重构整个认证系统，添加 OAuth，写测试，更新文档，
优化性能，添加速率限制"

# 好：分步骤
"第一步：只重构 src/auth/login.ts，把回调改为 async/await"
```

### 不提供上下文就要求修改

```
# 不好
"修复这个函数"（没有指定文件）

# 好
"修复 src/utils/date.ts 中的 formatDate() 函数，
当输入为 null 时应该返回空字符串而不是抛出异常"
```

### 要求 AI 猜测意图

```
# 不好
"你觉得这段代码有什么问题？"（太开放）

# 好
"审查 src/api/payments.ts，重点检查：
1. 是否有竞态条件
2. 错误处理是否完整
3. 是否有安全漏洞"
```

### 忽略 AI 的计划直接要求执行

```
# 不好的工作流
直接说："重构整个 auth 模块"
→ AI 开始大量修改
→ 发现方向不对，但已经改了很多文件

# 好的工作流
"先列出重构计划，等我确认后再执行"
→ 审查计划
→ 确认或调整方向
→ "按照计划执行第 1 步"
```

| 工具 | 触发深度思考 | 指定文件 | 只读参考 |
